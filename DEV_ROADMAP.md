## Phase 1 — 법적 필수 + 전환율 개선 (1~2주)

### 🔴 1. 개인정보처리방침 & 이용약관

> **왜 가장 먼저인가?**
> 현재 상태로 외부에 서비스를 공개하면 개인정보보호법(PIPA) 위반입니다.
> 회원의 이메일·이름·GitHub 계정 등을 수집하고 있으므로 법적 고지는 선택이 아닙니다.

**작성 방향 (AI 지원 활용)**

두 문서가 필요합니다:

```
1. 개인정보처리방침 (Privacy Policy)
   → 수집 항목, 목적, 보유 기간, 제3자 제공, 열람/삭제 권리 명시

2. 서비스 이용약관 (Terms of Service)
   → 서비스 이용 조건, 금지 행위, 책임 제한, 분쟁 해결
```

**핵심 작성 내용 (Side Project Mate 기준)**

```markdown
수집하는 개인정보:

- 필수: 이메일 주소, 비밀번호(암호화 저장), 닉네임
- 선택: 휴대폰 번호, GitHub 아이디, Solved.ac 아이디, 블로그 URL
- 자동 수집: 서비스 이용 기록, 접속 IP, 쿠키

수집 목적:

- 서비스 제공 및 회원 관리
- 팀 매칭 서비스 운영
- 고지사항 전달

보유 기간:

- 회원 탈퇴 시 즉시 삭제 (단, 관련 법령에 따라 일정 기간 보관)

제3자 제공:

- GitHub API: GitHub 연동 동의 시에만
- Cloudinary: 이미지 업로드 시
```

**구현 방법**

```
1. Claude에게 "Side Project Mate 서비스 기준으로 개인정보처리방침 초안 작성" 요청
2. 법무 AI 서비스(로앤컴퍼니 등) 또는 개인정보보호위원회 표준 양식 참고
3. src/app/privacy/page.tsx 신설
4. src/app/terms/page.tsx 신설
5. 푸터에 링크 연결
6. 회원가입 시 "이용약관 및 개인정보처리방침 동의" 체크박스 추가
```

> ⚠️ 완전한 법률 검토는 전문가에게 맡기되, AI로 초안을 작성하고 최소한의 법적 구색을 갖추는 것이 목표입니다.

### 🟠 2. 소셜 로그인 (GitHub OAuth + Google OAuth)

**왜 중요한가?**

개발자를 타겟으로 하는 서비스에서 GitHub 로그인 부재는 치명적입니다.
A/B 테스트 데이터에 따르면 소셜 로그인 추가만으로 **회원가입 전환율이 40~60% 향상**됩니다.

**구현 계획**

next-auth 기반이므로 Provider 추가만 하면 됩니다. 작업량이 매우 적습니다.

```typescript
// src/lib/auth.ts 에 추가
import GitHubProvider from 'next-auth/providers/github';
import GoogleProvider from 'next-auth/providers/google';

providers: [
  GitHubProvider({
    clientId: process.env.GITHUB_CLIENT_ID!,
    clientSecret: process.env.GITHUB_CLIENT_SECRET!,
  }),
  GoogleProvider({
    clientId: process.env.GOOGLE_CLIENT_ID!,
    clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
  }),
  // 기존 CredentialsProvider 유지
];
```

**처리해야 할 엣지 케이스**

```
1. 소셜 로그인 첫 가입 시 → OnboardingWizard 플로우로 연결 (기존 로직 재사용)
2. 같은 이메일로 일반 계정이 있을 때 소셜 로그인 시도 → 계정 연결 또는 충돌 처리
3. GitHub 소셜 로그인 시 GitHub 아이디 자동 입력 (프로필 연동 UX 개선)
```

**우선순위**: GitHub → Google 순서로 개발하세요. 타겟 유저에게 GitHub이 더 중요합니다.

### 🟡 4. 모바일 접근성 전략 — 칸반·WBS 대응

> 이 항목이 **가장 어려운 기술적 난제**입니다.
> 개발 리더 관점에서 "완벽한 모바일 대응"보다 **"모바일에서 망가지지 않는" 수준을 목표**로 잡는 것이 현실적입니다.

#### 원칙: 모바일에서 편집 기능은 포기, 조회 기능은 반드시 동작

| 화면               | 데스크탑         | 모바일 전략                      |
| ------------------ | ---------------- | -------------------------------- |
| 프로젝트 목록/상세 | 풀 기능          | ✅ 반응형 그대로                 |
| 프로필             | 풀 기능          | ✅ 반응형 그대로                 |
| 칸반 보드          | 드래그&드롭 편집 | ⬇️ 탭 전환 + 탭 터치로 상태 변경 |
| WBS 간트 차트      | 드래그 날짜 조정 | ⬇️ 심플 태스크 리스트 뷰로 교체  |
| 채팅               | 풀 기능          | ✅ 모바일 우선 UI                |

#### 칸반 보드 모바일 대응 방법

dnd-kit은 터치 이벤트를 지원하지만, 작은 화면에서 드래그는 UX가 나쁩니다.
모바일에서는 **탭으로 카드를 선택 → 상태 드롭다운으로 이동**하는 방식을 권장합니다.

```typescript
// 모바일 감지 후 분기 처리
const isMobile = useMediaQuery('(max-width: 768px)');

// 모바일에서는 DndContext 비활성화
// 대신 카드 클릭 시 상태 선택 모달 표시
{isMobile ? (
  <MobileKanbanView tasks={tasks} onStatusChange={handleStatusChange} />
) : (
  <DndContext ...>
    <KanbanBoard />
  </DndContext>
)}
```

**모바일 칸반 뷰 구조**

```
[전체] [To Do] [In Progress] [Done]  ← 탭 필터

카드 1: 기획서 작성
카드 2: API 설계          [상태 변경 ▾]
카드 3: UI 디자인
```

#### WBS/간트 차트 모바일 대응 방법

간트 차트는 **태블릿 이상에서만 표시**하고, 모바일에서는 태스크 리스트로 자동 전환합니다.

```typescript
// WBS 페이지 상단에 추가
const isMobile = useMediaQuery('(max-width: 768px)');

{isMobile ? (
  <MobileTaskList
    tasks={tasks}
    onTaskClick={(task) => setSelectedTask(task)}
  />
) : (
  <GanttChart tasks={tasks} />
)}
```

**모바일 태스크 리스트 UI**

```
📅 일정 관리                    [+ 추가]

▶ 기획 단계 (2개)
  □ 기획서 작성      3/1-3/5   김개발  [진행중]
  □ 와이어프레임     3/3-3/8   이디자인 [대기]

▶ 개발 단계 (1개)
  □ API 개발         3/8-3/20  김개발  [대기]
```

#### 실행 체크리스트

```
1. useMediaQuery 훅 신설 (src/hooks/useMediaQuery.ts)
2. 칸반 보드: MobileKanbanView 컴포넌트 신설
3. WBS: MobileTaskList 컴포넌트 신설
4. 공통 레이아웃(Header, Footer, 프로젝트 목록): Tailwind 반응형 클래스 점검
5. Chrome DevTools 모바일 시뮬레이터로 주요 페이지 전수 검증
```

## Phase 3 — 수익화 + 성장 (1~2개월)

### 💬 5. 광고 도입 — UX 영향 분석 및 권고안

> **결론부터**: 광고를 넣어도 됩니다. 단, **어디에 넣느냐**가 서비스 성패를 가릅니다.

#### 광고 배치 원칙

| 영역                      | 광고 허용 여부   | 이유                                   |
| ------------------------- | ---------------- | -------------------------------------- |
| 랜딩/홈 페이지            | ✅ 가능          | 비로그인 방문자가 주 대상              |
| 프로젝트 목록 페이지      | ✅ 가능          | 탐색 중 자연스러운 노출                |
| 프로젝트 상세 페이지 하단 | ✅ 가능          | 콘텐츠 소비 후 노출                    |
| 개인 프로필 페이지        | ⚠️ 주의          | 타인 프로필에만 허용, 본인 프로필 제외 |
| **칸반 보드 / WBS**       | ❌ **절대 금지** | 집중 업무 공간 침해 → 즉시 이탈        |
| **채팅 페이지**           | ❌ **절대 금지** | 대화 흐름 방해 → 최악의 UX             |
| **대시보드 내부**         | ❌ **절대 금지** | 협업 도구에서 광고는 신뢰 손상         |

#### 권장 광고 전략 (단계별)

**1단계: Google AdSense 또는 카카오 AdFit (초기)**

```
- 위치: 홈 페이지 하단 배너, 프로젝트 목록 사이드바 또는 하단
- 형식: 반응형 배너 (728x90 데스크탑, 320x50 모바일)
- 장점: 설치 쉬움, 자동 최적화
- 단점: RPM(1000 노출당 수익)이 낮음 (~$1~5)
```

**2단계: 개발자 특화 광고 네트워크 (성장 후)**

```
- Carbon Ads (개발자 대상 최적화, RPM ~$10~30)
- Ethical Ads (개인정보 미추적, 개발 커뮤니티 특화)
- 국내: Naver 성과형 광고, 직접 협찬 제안
```

**3단계: 네이티브 스폰서십 (MAU 5,000 이상)**

```
- 개발 교육 플랫폼 (인프런, 패스트캠퍼스) 배너
- SaaS 도구 (Notion, Linear, GitHub Copilot) 스폰서
- 클라우드 서비스 (AWS, Vercel) 스타트업 크레딧 프로그램
```

#### UX 가이드라인 (광고 넣을 때 지켜야 할 것)

```
1. 광고는 콘텐츠와 시각적으로 구분 ("광고" 레이블 명시)
2. 전체 페이지에서 광고 영역이 20%를 초과하지 않도록
3. 자동 재생 광고, 팝업 광고는 절대 금지
4. 로그인한 유료 사용자에게는 광고 숨김 처리 (Freemium 전환 동기 부여)
```

#### Freemium 전환 방향 (순서)

```
[현재] 완전 무료
  ↓ (MAU 500명 이상)
[1단계] 무료 + 광고 / 유료($9,900/월) 광고 없음 + 부가 기능
  ↓ (MAU 2,000명 이상)
[2단계] 기능 제한 도입 (무료: 프로젝트 3개, 팀원 5명 / 유료: 무제한)
  ↓ (MAU 5,000명 이상)
[3단계] 팀 플랜 도입 ($29,900/월, 10인 이상 팀)
```

> **중요**: 기능 제한 도입은 초기에는 하지 마세요. MAU가 일정 수준에 도달하기 전에 기능 제한을 걸면 신규 유저 유입이 막힙니다.

### 🟡 6. 팀원 리뷰 & 추천 알고리즘

**구현 순서 권고**: 리뷰 시스템 → 추천 알고리즘 (데이터 없으면 추천 불가)

#### 리뷰 시스템 (먼저 구현)

이미 Comment 모델이 있으므로 확장하는 방향으로 구현합니다.

```typescript
// 새 Review 모델 구조 (src/lib/models/Review.ts)
{
  projectId: ObjectId,
  reviewerId: ObjectId,    // 리뷰 작성자
  revieweeId: ObjectId,    // 리뷰 대상자
  rating: Number,          // 1~5점
  tags: [String],          // "책임감있어요", "소통잘해요", "실력있어요"
  comment: String,
  isPublic: Boolean,
  createdAt: Date
}
```

**트리거**: 프로젝트 상태가 "완료"로 변경될 때 팀원들에게 리뷰 요청 알림 발송

#### 추천 알고리즘 (리뷰 데이터 쌓인 후 구현)

```
매칭 점수 = (기술 스택 일치도 × 0.4)
           + (가용 시간 겹침률 × 0.3)
           + (프로젝트 카테고리 관심도 × 0.2)
           + (리뷰 평점 × 0.1)
```

### 🟡 7. 커뮤니티/게시판 — 도입 여부 판단 기준

**도입 추천 조건** (이 중 1개 이상 해당되면 도입)

- [ ] MAU 1,000명 이상인데 재방문율이 30% 미만인 경우
- [ ] 채팅 이외에 유저들이 서로 소통하려는 시도가 보이는 경우
- [ ] 경쟁 서비스가 커뮤니티 기능으로 유저를 뺏어가는 경우

**도입하지 않아도 되는 경우**

- 핵심 기능(팀 매칭, 협업 도구)이 아직 완성되지 않은 경우
- 운영 인력이 부족해 스팸 관리가 어려운 경우

**현재 판단**: 다른 P0~P2 작업을 완료한 후 MAU 지표를 보고 결정하세요.
모델(Post, Comment)이 이미 있으므로 필요 시 빠르게 붙일 수 있습니다.

## 마지막으로 — 개발 리더 관점의 한마디

이 프로젝트는 **기술적 완성도는 이미 출시 가능한 수준에 근접**해 있습니다.
남은 과제는 기술보다 **비기술적 요소** (법적 고지, 소셜 로그인 UX, 랜딩 설득력)에 집중되어 있습니다.

가장 흔한 실수는 "완벽해질 때까지 출시를 미루는 것"입니다.
**채팅 완료 후 개인정보처리방침까지만 갖추면 외부 베타 공개가 가능합니다.**
실제 유저의 피드백 없이 개발 방향을 잡는 것은 비효율적입니다.

```
권장 출시 시나리오:
[Phase 1 완료] → 소규모 베타 공개 (지인 50~100명)
[Phase 2 완료] → 개발자 커뮤니티 공개 (OKR 설정)
[Phase 3 완료] → 정식 런칭 + 광고 도입
```

> 작은 완성 → 공개 → 피드백 → 개선의 사이클이 가장 빠른 성장 경로입니다.
