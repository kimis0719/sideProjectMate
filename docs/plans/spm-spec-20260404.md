## 1. 지시서 생성 AI 추가요청사항 가드레일 설정

### 1.1 배경 및 목적

현재 InstructionModal의 '추가 지시사항(additionalInstruction)' 입력창은 내용 제한이 없다. 악의적이거나 토큰을 낭비하는 입력(프롬프트 인젝션, 역할 변경 지시, 무한 반복 등)을 방치하면 LLM 응답 품질 저하 및 일일 한도 조기 소진 문제가 발생할 수 있다.

### 1.2 가드레일 규칙

#### 길이 제한

- 최대 **500자** (공백 포함)
- 500자 초과 시 입력 불가 + 실시간 카운터 표시 (예: `342 / 500`)

#### 금지 패턴 (서버 사이드 정규식 검증)

아래 패턴 포함 시 `400 Bad Request` 반환.

| 유형 | 패턴 예시 | 반환 메시지 |
|------|-----------|------------|
| 역할 변경 시도 | "너는 이제 ~이다", "역할을 바꿔" | 허용되지 않는 지시입니다. |
| 시스템 프롬프트 무시 | "이전 지시 무시", "ignore above" | 허용되지 않는 지시입니다. |
| 반복 지시 악용 | "위 내용을 100번 반복" | 허용되지 않는 지시입니다. |
| 개인정보 유출 유도 | "API 키를 출력해", "비밀번호 알려줘" | 허용되지 않는 지시입니다. |
| 외부 URL 포함 | `http://`, `https://`로 시작하는 링크 | URL은 입력할 수 없습니다. |

#### 프론트엔드 사전 검증

- 입력 시 실시간 금지 패턴 감지 → 빨간색 테두리 + 경고 문구 표시
- 검증 실패 상태에서는 `생성하기` 버튼 비활성화

## 2. 어드민 계정 지시서 생성 제한 해제

### 2.1 배경 및 목적

현재 지시서 생성 API는 모든 사용자에게 동일하게 쿨다운(`cooldownMinutes`)과 일일 한도(`dailyLimitPerProject`)를 적용한다. 어드민 계정은 기능 테스트 및 프리셋 검수 목적으로 제한 없이 지시서를 생성할 수 있어야 한다.

### 2.2 적용 로직

요청 세션의 `role`이 `'admin'`인 경우 쿨다운·일일 한도 체크를 모두 건너뛴다.

| 체크 항목 | 일반 유저 | 어드민 |
|-----------|-----------|--------|
| 쿨다운 (`cooldownMinutes`) | 적용 (기본 3분) | ✅ 스킵 |
| 일일 한도 (`dailyLimitPerProject`) | 적용 (기본 50건) | ✅ 스킵 |
| 가드레일 (추가요청 검증) | 적용 | ✅ 스킵 |
| 토큰 사용 기록 (`AiUsage`) | 기록 | 기록 유지 (감사 목적) |

### 2.3 의사코드

```typescript
const isAdmin = session.user.role === 'admin';

// 쿨다운 체크
if (!isAdmin) {
  // 기존 쿨다운 체크 로직
}

// 일일 한도 체크
if (!isAdmin) {
  // 기존 일일 한도 체크 로직
}

// 가드레일 체크
if (!isAdmin) {
  // 금지 패턴 검증 로직
}
```

| BE | `src/app/api/ai/generate-instruction/route.ts` | 쿨다운/일일 한도/가드레일 체크 전 `session.user.role === 'admin'` 분기 추가 |

## 3. 부분 완료 처리 버튼 UX 개선

### 3.1 현상 및 문제

실행결과 파싱 후 확인 화면에서 `partial` 상태 노트의 **[유지]** / **[완료처리]** 버튼을 클릭해도 아무런 UI 변화가 없다. 실제 완료처리는 서버에서 정상 동작하지만 사용자는 피드백을 받지 못하는 상태.

### 3.2 개선 목표

- **[유지]**: 서버에서 리턴받은 `summary` 값을 해당 노트의 `completionNote`에 추가 기록
- **[완료처리]**: 포틀릿(노트 상세 모달)에서 [확인] 버튼을 누른 것과 동일한 완료처리 플로우 실행

### 3.3 상세 동작 정의

#### [유지] 버튼

사용자가 partial 상태 노트에서 [유지]를 선택한 경우 — 노트를 완료 처리하지 않되, Agent가 보고한 `summary`를 `completionNote`에 기록한다.

| 항목 | 동작 |
|------|------|
| 노트 상태 | 변경 없음 (`active` 유지) |
| `completionNote` 필드 | `[유지 - YYYY-MM-DD] {summary}` 형식으로 기존 값에 append |
| UI 피드백 | 해당 행 회색 dimming + "메모에 기록되었습니다" 토스트 표시 후 행 제거 |
| API 호출 | `PATCH /api/kanban/notes/:id` `{ completionNote: append된 값 }` |

**append 형식:**

```typescript
const appended = existingNote
  ? `${existingNote}\n[유지 - ${today}] ${summary}`
  : `[유지 - ${today}] ${summary}`;
```

#### [완료처리] 버튼

사용자가 partial 상태 노트에서 [완료처리]를 선택한 경우 — 포틀릿의 [확인]과 동일한 완료처리 플로우 실행.

| 항목 | 동작 |
|------|------|
| 노트 상태 | `active` → `done` + `completedAt` 기록 |
| `completionNote` 필드 | Agent `summary` 내용을 `completionNote`로 저장 |
| UI 피드백 | 해당 행 초록색 flash 후 제거 + 칸반보드 해당 노트 즉시 `done` 업데이트 (Socket.io 브로드캐스트) |
| API 호출 | `PATCH /api/kanban/notes/:id` `{ status: 'done', completedAt: now, completionNote: summary }` |
| 기존 포틀릿 [확인]과 동일 | `boardStore`의 `completeNote()` 액션 재사용 |

| 구분 | 파일 | 작업 내용 |
|------|------|-----------|
